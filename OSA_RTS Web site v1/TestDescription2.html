TSF Libraries

ATML Test Description allows signals to be fully defined using the IEEE Std. 1641 Signal Modeling Standard. This allows signals to be defined in a Signal modeling libraries canned Test Signal Framework (TSF) libraries. Any signal can make use of the signals contained within a TSF Library mealy by referencing the TSF Library schema and the TSF within it.

What is a Test Signal Framework model?

  • A model of a signal required to test a UUT
  • It is a shorthand way of defining a complicated signal
  • Can comprise a single BSC
  • Can be a complex arrangement of BSCs
  • Can include previously defined TSF models
  • Its "Interface Properties" are the interface to the Test Requirement

Why do we need TSF libraries?

  • They contain all the signals required by a specific technology, environment or application
  • 1641 has an example library (corresponding to C/ATLAS)
  • They include simple TSF models such as AC_SIGNAL and DC_SIGNAL
  • They include complex TSF models such as SYNCHRO and TACAN
  • Can include previously defined TSF models
  • Its "Interface Properties" are the interface to the Test Requirement

There are multiple ways Test Description may need to reference a TSF Library each refers to the same libraries but are used by different applications

xsi:schemaLocation TsfLibraries

TSF Libraries generally come with their own documentation (usually generated from the TSF Library XML. In the case of the ATML Demo program the TSF libraries available are:

  • STDTSFLib (IEEE Std 1641 (716 Standard Library))
  • ATMLDemo2DigitalPatterns - Defines the parallel digital patterns used during Phase II on the digital IO
  • OnedBCompressionTest - Complex RF Test methods for testing RF UUTs used as part of Phase 2 test program
  • ATMLDemoRTCASSLib - Simplified UUT TSF Library for teh ATML Demo UUT. Only used during Phase I test program to interface with RTCASS ATE

xsi:schemaLocation

In order to validate the use of any TSF the schema location of the TSF Library XMLSchema(.xsd) file must be declared. Different applications and validators can do this in a variety of ways however the use of the xsi:schemaLocation is a classic way by which the location of the TSF Library schema can be given to the Test Description to allow XML validation to proceed.

Note: multiple declarations of xsi:schemaLocation may not work with all valdiators, its best to ensure that the top level declaration of schemaLocations contains all the locations.

TsfLibraries

Provides a way for Test Description to identify the specific libraries it's using, and also to identify the locations TSF library schema and the TSF library 'xml', which may be in different places Generally TsfLibraries should be completed as it helps identify extra information as well as which XMLSchemas refer to only TSF Libraries.

- <td:TsfLibraries>
- < td:TsfLibrary name ="urn:IEEE-1641:2010:STDTSFLib" ID ="1">
  <td:XmlSchemaURL>STDTSFLib.xsd</td:XmlSchemaURL>
  <td:XmlInstanceDocumentURL>STDTSFLib.xml</td:XmlInstanceDocumentURL>
  </td:TsfLibrary>
- < td:TsfLibrary name ="ATMLDemo2DigitalPatterns" ID ="2">
  <td:XmlSchemaURL>ATMLDemo2DigitalPatterns.xsd</td:XmlSchemaURL>
  <td:XmlInstanceDocumentURL>ATMLDemo2DigitalPatterns.xml</td:XmlInstanceDocumentURL>
  </td:TsfLibrary>
- < td:TsfLibrary name ="OneDBCompressionTest" ID ="3">
  <td:XmlSchemaURL>OneDBCompressionTest.xsd</td:XmlSchemaURL>
  <td:XmlInstanceDocumentURL>OneDBCompressionTest.xml</td:XmlInstanceDocumentURL>
  </td:TsfLibrary>
  </td:TsfLibraries>
